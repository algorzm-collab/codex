# STRAT Core Website Architecture v2

## 1. 핵심 원칙

STRAT 홈페이지의 본질은 컨설팅 회사의 신뢰 구조를 유지하면서, 첫 진입 경험을 고객 문제 중심으로 바꾸는 것이다.

기본 속성:

- 회사소개
- 구성원 소개
- 관련 실적
- 문의하기
- 내 문제 유형 사전진단

이 5개는 서로 경쟁하지 않는다. 역할이 다르다.

```text
내 문제 유형 사전진단 = 후킹 엔진
관련 실적 = 신뢰 증거
회사소개 = 정체성
구성원 소개 = 사람 신뢰
문의하기 = 전환
```

## 2. 추천 사용자 흐름

전통 컨설팅 홈페이지:

```text
회사소개 -> 서비스소개 -> 실적 -> 구성원 -> 문의
```

STRAT형 그로스 컨설팅 홈페이지:

```text
내 문제 짚어보기
-> 문제 이름 붙이기
-> 관련 실적 보기
-> 회사/구성원 신뢰 확인
-> 문의하기
```

즉, 기본 홈페이지 구조는 유지하되 첫 입구를 “회사 설명”이 아니라 “고객 문제 인식”으로 바꾼다.

## 3. GNB 구조

권장 메뉴:

```text
문제 짚어보기 | 실적 | 서비스 | 회사소개 | 구성원 | 인사이트 | 문의하기
```

영문/혼합형:

```text
Issue Finder | Cases | Services | Company | People | Insights | Contact
```

상단 고정 CTA:

```text
우리 문제 짚어보기
```

## 4. 홈 페이지 최종 섹션 구조

### 1. Hero: 문제 후킹

목적:

- 방문자가 자기 문제를 느끼고 첫 선택을 하게 만든다.

카피:

> 보고서는 있는데 실행이 안 되나요?  
> 몇 가지 선택으로 조직의 진짜 병목을 짚어드립니다.

보조 카피:

> STRAT은 짧은 선택을 바탕으로 현재 문제가 전략, 조직, 성과관리, 인사, ESG/AI 중 어디에 가까운지 추론하고, 유사 실적과 가설적 접근을 제시합니다.

CTA:

- 우리 문제 짚어보기
- 관련 실적 보기

### 2. Issue Finder: 내 문제 유형 사전진단

목적:

- 설문이 아니라 문제 추론 경험 제공

질문:

- 요즘 조직에서 가장 자주 들리는 말은 무엇인가요?
- 그 문제가 실제로 터지는 장면은 어디인가요?
- 지금 필요한 것은 어느 쪽에 더 가깝나요?

결과:

- 문제 명명
- 왜 그렇게 보이는지
- 관련 실적 가설
- 가설 솔루션
- STRAT 문의 CTA

### 3. Problem Mirror: 문제 공감

목적:

- 방문자가 스크롤하면서 “맞다, 우리 얘기다”라고 느끼게 한다.

문장:

- 보고서는 있는데 실행이 안 된다.
- 이 일이 어느 부서 책임인지 애매하다.
- KPI는 많은데 무엇을 바꿔야 할지 모르겠다.
- 평가제도는 있는데 구성원들이 납득하지 못한다.
- 조직개편 이야기는 많은데 기준이 없다.
- AI/ESG 과제가 생겼는데 실행과제로 정리되지 않았다.

### 4. Related Cases: 관련 실적

목적:

- 문제 추론 결과를 신뢰로 전환한다.

사례 카드 구조:

```text
고객 유형
문제
STRAT 접근
산출물
관련 문의 CTA
```

중요:

실적은 단순 목록이 아니라 “비슷한 문제를 해결해본 증거”로 보여준다.

### 5. Services: 서비스

목적:

- STRAT이 어떤 문제를 어떤 솔루션 체계로 다루는지 보여준다.

7대 클러스터:

- 전략체계 정렬
- KPI 및 성과관리체계
- 조직구조 및 R&R
- 업무프로세스 및 협업
- 인사·직무·역량
- ESG 및 사회적가치
- AI/DT 전환

### 6. Company: 회사소개

목적:

- STRAT이 어떤 회사인지, 어떤 관점으로 일하는지 보여준다.

내용:

- 회사 철학
- 전문 분야
- 수행 방식
- 고객 유형
- 차별점

카피 방향:

> STRAT은 전략을 보고서로 끝내지 않고, 조직과 성과체계로 연결하는 실행 중심 전략자문그룹입니다.

### 7. People: 구성원 소개

목적:

- 컨설팅은 결국 사람에 대한 신뢰이므로, 누가 문제를 다루는지 보여준다.

구성:

- 대표/파트너
- 분야별 전문가
- 전문 영역
- 대표 수행 과제
- 한 줄 관점

좋은 구성원 소개:

```text
이름 / 직함
전문 분야
핵심 경력 3개
대표 프로젝트 유형
한 줄 소개
```

### 8. Method: 방법론

목적:

- 전문성과 일관성을 보여준다.

6단계:

- Issue Framing
- Evidence Mapping
- Strategy Design
- Operating Model
- Execution Roadmap
- Decision Package

### 9. Insights: 인사이트

목적:

- SEO, 재방문, 신뢰 형성

콘텐츠:

- 중장기전략 착수 전 체크리스트
- KPI가 전략과 분리되는 이유
- R&R 재정립이 조직개편보다 먼저 필요한 경우
- 성과평가제도가 성과를 만들지 못하는 이유

### 10. Contact: 문의하기

목적:

- 최종 전환

문의 문구:

> 홈페이지 결과는 간단한 선택을 바탕으로 한 가설입니다. 실제 과제 범위와 접근 방식은 STRAT과의 상담에서 구체화합니다.

CTA:

- STRAT에 문의하기
- 유사 과제 상담 요청
- 우리 조직도 해당되는지 물어보기

## 5. 페이지별 역할

### Issue Finder

역할:

- 방문자의 문제를 가볍게 추론한다.
- 상담 전환의 후킹 역할을 한다.

주의:

- 설문조사처럼 보이면 안 된다.
- 자료 제출을 요구하면 안 된다.

### Cases

역할:

- 실적을 회사 자랑이 아니라 문제 해결 증거로 보여준다.

### Company

역할:

- STRAT의 관점과 정체성을 설명한다.

### People

역할:

- 컨설팅 수행자의 신뢰를 만든다.

### Contact

역할:

- 사용자가 더 깊은 논의를 위해 회사로 오게 한다.

## 6. 최종 판단

“내 문제 유형 사전진단”은 홈페이지의 본질을 흐리는 장치가 아니다.  
오히려 회사소개, 구성원소개, 실적, 문의하기로 이어지는 경로를 더 강하게 만드는 입구다.

최종 구조:

```text
문제 후킹
-> 문제 추론
-> 유사 실적
-> 회사 신뢰
-> 구성원 신뢰
-> 문의
```
