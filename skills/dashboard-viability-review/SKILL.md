# Dashboard Viability Review

## Overview

Use this skill to prevent dashboard work from becoming performative UI. The dashboard must reduce CEO friction and increase operational continuity, or it should be simplified or stopped.

## Decision Workflow

1. Identify the source of truth.
   - Continue only if durable state already exists in GitHub issues, commits, sheets, files, or another reliable system.
   - Stop or simplify if the dashboard itself is expected to become the primary memory.
   - Prefer GitHub Issues for task/idea/bug tracking, pull requests for proposed changes and checks, and workflow artifacts for generated run outputs.

2. Classify the dashboard role.
   - Read-only cockpit: usually worth continuing.
   - Command surface: continue only after read-only monitoring is reliable.
   - Approval/control system: defer unless auth, audit log, rollback, and human confirmation rules are explicit.

3. Check failure factors.
   - Manual duplicate entry
   - No durable handoff path
   - No clear user decision the dashboard supports
   - Live sync before static evidence is reliable
   - UI polish replacing workflow proof
   - Hidden auth, token, or permission dependency

4. Choose one decision.
   - `continue`: existing loop is reliable and the next change is small.
   - `simplify`: dashboard is useful, but scope is too broad.
   - `stop`: it adds maintenance burden without improving decisions.

## Output Requirements

Always report:

- Decision: continue, simplify, or stop
- Evidence
- Failure factors learned
- Smallest next action
- What not to build yet

## Default Recommendation

Prefer a static, evidence-backed cockpit first. Add live sync only after one read-only source path is reliable. Add remote approval/control last.

## GitHub-Based Minimal Evidence Loop

Use this loop when token budget is tight:

1. Read the relevant GitHub issue or queue item.
2. Inspect only the files named by the issue or the smallest local dashboard surface.
3. Make the smallest change that improves decision quality.
4. Run the nearest smoke check or syntax check.
5. Preserve state with a branch, commit, PR/comment, or explicit handoff.

Do not add a dashboard feature unless it maps to one of these evidence sources:

- Issue state: what work exists and why
- Pull request state: what changed, what checks say, and what blocks merge
- Artifact state: what output was generated by automation
- Handoff state: what the next agent should do

## Failure Lessons To Capture

When dashboard work stalls, record the reusable lesson in the skill or dashboard data:

- Missing durable source of truth
- Too much live orchestration before read-only sync
- CEO required to copy status between tools
- Token spent on UI instead of handoff continuity
- Approval flow without audit trail or rollback
